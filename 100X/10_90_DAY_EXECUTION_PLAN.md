# 90-day execution plan

## Days 0–7 — trustworthy engineering slice
Production schema, state machine, OIDC skeleton, secure uploads, preserved originals, candidate extraction, reviewer gate, public dossier, report/RTI export and complete synthetic smoke path.

## Days 8–21 — hardened staging
Managed PostgreSQL/storage, migrations, quarantine/scanning, redaction, public projection, correction versions, observability, backup/restore, accessibility, load/security tests and signed release candidate.

## Days 22–30 — controlled pilot
One partner, one narrow infrastructure class, two-person review, no automated public crawling, documented source scope and daily incident readiness. Release as beta unless all P0 gates pass.

## Days 31–45 — repeatability
Second and third partners, connector SDK, CPPP/manual source adapters, benchmark corpus, reviewer analytics, OC4IDS export validation and improved onboarding.

## Days 46–60 — proof of value
Five completed workflows, measured time savings, public methodology, transparent failures, correction exercise, authority response workflow and first external contributor.

## Days 61–90 — v1 decision
Security review, multi-tenant boundary if required, second language benchmark, governance, sustainable operating budget and go/no-go against adoption gates.

## Kill/pivot criteria at day 90
Pivot or stop if fewer than three teams complete a workflow, time savings are below 20%, reviewers distrust extraction, source access cannot be maintained lawfully, or operating/legal risk lacks an accountable owner.
